# Fork Notes: HDR AVIF Preview Bypass

This branch is a small custom fork for HDR AVIF originals whose Immich previews and thumbnails look flat or incorrectly colored after the normal Sharp/libvips pipeline. The goal is to preserve HDR/color metadata for generated preview assets without changing the behavior of other image types.

## User-Facing Behavior

- Adds an admin image setting named "AVIF HDR bypass".
- When enabled, only HDR AVIF originals generate AVIF previews and thumbnails through the custom bypass path.
- The normal preview and thumbnail format dropdowns still control all other images.
- Existing HDR AVIF preview/thumbnail files are backfilled once when the setting is changed from disabled to enabled.

## Safety Boundaries

- No database migrations.
- No new tables or columns.
- No direct deletion or manual invalidation of existing preview records.
- The targeted backfill query is read-only and only queues normal Immich thumbnail jobs.
- Old preview/thumbnail files are replaced through Immich's existing thumbnail generation and `syncFiles` path after new files are generated successfully.
- The custom converter only runs for output paths ending in `_preview.avif` or `_thumbnail.avif`.
- The converter writes to a temporary file first and renames into place only after encoding succeeds.
- If the custom converter fails, thumbnail generation falls back to the existing Sharp path.

## Custom Pipeline

The custom AVIF pipeline is isolated in:

- `server/src/utils/avif-hdr-bypass.ts`

It uses `ffprobe` to read AVIF color metadata, `ffmpeg` to downscale pixels, and `avifenc` to encode the preview/thumbnail while preserving CICP/range and, when present, content light level metadata.

The runtime dependency for this is:

- `libavif-bin`

It is installed in the server Docker image to provide `avifenc` and `avifdec`.

## Backfill

When `image.avifHdrBypass` changes from `false` to `true`, the media service queues `AssetGenerateAvifHdrThumbnailsQueueAll`.

That job scans for image assets where:

- the original filename ends in `.avif`
- the asset appears HDR according to the same metadata rules used by thumbnail generation
- the preview or thumbnail is missing or is not already `.avif`

Matching assets are queued through the existing `AssetGenerateThumbnails` job.

## Stock Immich Rollback

This branch intentionally avoids schema changes so a stock Immich build can be restored without database repair. Stock Immich should ignore the extra config key as an unknown setting. Any AVIF preview/thumbnail files generated by this branch are ordinary `asset_file` preview/thumbnail rows and can be replaced by stock Immich by running normal thumbnail regeneration with stock image settings.

## iOS Mobile App Compatibility

The current HDR AVIF previews work in supported web browsers but do not work in the Immich iOS app. The app displays only its blurred placeholder and the image cannot be zoomed.

### Failed YUV420 Experiment

On 2026-07-13, branch `app-hdr-previews` tested whether Apple's app-side decoder rejected the original YUV444 chroma format. The test changed only the FFmpeg intermediate pixel format from `yuv444p10le` to `yuv420p10le`.

The controlled test used `DSCF1090-2.avif` and produced:

- a complete `1440x960` preview and `250x167` thumbnail
- 10-bit YUV420
- full range
- BT.2020 primaries and matrix
- PQ (`smpte2084`) transfer characteristics
- an HTTP `Content-Type` of `image/avif`

The Immich iOS app still showed the same blurred, non-zoomable placeholder. Therefore, **do not retry YUV420 chroma subsampling as a standalone fix**.

### Why a Server-Only HDR Fix Is Insufficient

The iOS app does not use Safari's AVIF rendering path. `mobile/ios/Runner/Images/RemoteImagesImpl.swift` downloads remote images with `URLSession`, decodes them through `CGImageSourceCreateThumbnailAtIndex`, and converts them to 8-bit device RGB with vImage. `mobile/lib/infrastructure/loaders/image_request.dart` then exposes those pixels to Flutter as `rgba8888`.

Even if an HDR AVIF decodes successfully, that 8-bit bridge cannot preserve its HDR pixel values. The realistic next directions are:

- serve an SDR JPEG/WebP fallback to the `immich-ios/*` user agent, which restores viewing and zooming but not HDR
- add an app-side native HDR still-image renderer, which is required for actual HDR display

## Merge Notes

Keep the custom AVIF work isolated. If upstream changes thumbnail generation, prefer adapting only the hook points in `media.service.ts` and `media.repository.ts` while keeping the AVIF-specific command logic in `server/src/utils/avif-hdr-bypass.ts`.

Do not expand the bypass to fullsize images unless that is an explicit future goal. The current branch is deliberately limited to previews and thumbnails.
